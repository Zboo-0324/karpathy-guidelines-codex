# Karpathy Guidelines for Codex

这是一个面向 Codex 的轻量级 skill，用来在写代码、审查代码、调试、重构或简化实现时，引导模型采用更紧凑、显式、可读、容易调试的工程风格。

它改编自公开的 Karpathy Guidelines 风格理念，并针对 Codex 的 skill 结构整理成可安装、可触发、可复用的说明文件。

## 这个 Skill 的作用

`karpathy-guidelines` 的核心目标不是强制某一种架构，而是在具体工程任务中持续提醒 Codex 优先选择简单、直接、容易理解的实现。

它会让 Codex 更倾向于：

- 保持控制流简单，避免不必要的深层嵌套和过度抽象。
- 优先写可读代码，而不是为了复用过早抽取 helper。
- 让状态、输入、输出和数据流动保持显式。
- 写出便于断点调试和局部推理的代码。
- 优先沿用项目已有模式，而不是引入新框架或新依赖。
- 对失败路径、边界条件和错误处理给出明确行为。

这个 skill 适合用作编码过程中的风格约束和 review checklist，尤其适用于需要降低复杂度、清理实现、修复问题或审查代码质量的任务。

## 适用场景

推荐在以下任务中使用：

- 编写新功能时，希望实现保持小而清晰。
- 修复 bug 时，希望先理解最小问题路径，再做精确改动。
- 重构代码时，希望避免把简单逻辑改成更复杂的抽象。
- 做 code review 时，希望重点检查可读性、数据流、失败模式和局部可推理性。
- 简化已有实现时，希望删除不必要的 clever code、死路径或模糊 helper。

不建议把它当作绝对规则使用。项目约定、用户明确要求、正确性和测试要求始终优先。

## 使用方法

在支持 Codex skills 的环境中安装该 skill 后，可以通过下面的方式显式触发：

```text
Use $karpathy-guidelines while writing, reviewing, debugging, or refactoring code.
```

也可以在具体请求里直接提到：

```text
使用 $karpathy-guidelines 帮我重构这个函数。
```

```text
用 karpathy-guidelines 的风格审查这段代码。
```

本仓库的 `agents/openai.yaml` 已配置默认提示词：

```yaml
default_prompt: "Use $karpathy-guidelines while writing, reviewing, debugging, or refactoring code."
```

并开启了隐式调用：

```yaml
allow_implicit_invocation: true
```

这表示在合适的编码、审查、调试或重构场景下，系统可以自动考虑使用这个 skill。

## 工作方式

使用该 skill 时，Codex 会优先遵循一个简单的工作流程。

编辑前：

- 找到需要改变的最小行为或代码路径。
- 先阅读周围代码，沿用项目现有风格。
- 如果任务会改变行为，优先使用项目已有测试流程验证。

编辑中：

- 做最小但完整的改动。
- 用清晰命名表达意图，少用解释显而易见代码的注释。
- 在请求目标达成后停止，避免顺手扩大重构范围。

完成前：

- 重新阅读 diff，检查是否引入了不必要的复杂度。
- 删除偶然产生的 clever code、死代码和模糊命名。
- 用最窄但有意义的命令验证结果，必要时再扩大验证范围。

## 核心准则

### 1. 保持控制流简单

优先使用直线式代码、小函数、显式分支和局部推理。除非项目已有模式确实依赖，否则避免元编程、深层嵌套、过度框架魔法或复杂控制流。

### 2. 先可读，再复用

不要只因为两段代码看起来相似就抽 helper。只有当抽象能提供更好的命名、隔离真实概念，或者减少已经造成风险的重复时，才值得抽取。

### 3. 让状态和数据流可见

输入、输出、状态变化和数据转换应尽量显式。避免隐藏全局状态、远距离副作用、隐式注册表和难追踪的数据修改。

### 4. 写容易调试的代码

当中间变量能帮助理解时，就给它一个明确名字。避免把重要逻辑塞进一行链式调用或多层嵌套表达式里。

### 5. 使用项目已有模式

优先使用仓库里已经存在的技术栈、工具函数和组织方式。只有在新依赖或新架构能明显减少复杂度时才引入。

### 6. 明确失败行为

在数据进入系统的边界处检查异常情况。错误应尽量清楚，避免宽泛捕获、静默失败或含糊默认值。

## Review Checklist

在完成任务前，可以用下面的问题检查实现质量：

| 问题 | 如果答案是否定的 |
| --- | --- |
| 读者能不跳转很多文件就理解主路径吗？ | 内联逻辑或改善边界。 |
| 输入、输出和状态变化是否可见？ | 让依赖和数据流更显式。 |
| 每个抽象是否承载了真实概念？ | 删除或合并空泛抽象。 |
| 失败模式能从错误、日志或测试中看清吗？ | 补充明确处理或覆盖。 |
| 改动是否符合项目本地约定？ | 对齐项目风格，除非有明确理由。 |

## 常见误用

- 把 guidelines 理解成禁止所有抽象。好的抽象仍然有价值，前提是它让系统更小、更清楚。
- 把项目已有风格改成个人偏好的风格。局部一致性比审美偏好更重要。
- 用泛化 helper 隐藏复杂度。如果 helper 名字很空泛，抽象本身通常也很空泛。
- 因为是“清理代码”就跳过验证。重构同样需要验证，因为行为不应该被意外改变。
- 过度注释显而易见的代码。优先用命名和结构表达意图，注释主要解释原因和约束。

## 指令优先级

当不同要求发生冲突时，优先级如下：

1. 用户的明确请求。
2. 当前仓库和项目的既有约定。
3. 安全性、正确性和测试要求。
4. `karpathy-guidelines` 的风格建议。

也就是说，这个 skill 是风格和工程判断辅助，不会覆盖用户明确指定的架构、项目规范或正确性要求。
